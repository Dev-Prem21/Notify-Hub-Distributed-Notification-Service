🏗️ System Overview

NotifyHub is a distributed notification system that demonstrates independent failure handling with automatic retries and dead-letter queues. When a user registers, the system publishes events to three separate notification channels that process independently.

Core Components

Responsibilities:
  • Receive user registration requests
  • Create users in-memory or database
  • Publish user.events to Kafka
  • Return 201 Created to client
Responsibilities:
  • Router: Routes events to 3 topics
  • 3 Consumers: Email, SMS, Push
  • Retry Logic: 3 attempts per service
  • DLQ Handler: Manages failures

🔄 High-Level Flow

Here's the big picture of how a message travels through the system:

1. CLIENT REQUEST
   ↓
   POST /api/users
   {id, name, email, phoneNumber}
   ↓

2. USER SERVICE PROCESSING
   ├─ Validate input
   ├─ Create user (in-memory)
   ├─ Publish UserRegisteredEvent
   │  └─ Topic: user.events
   │  └─ Message: {userId, email, phoneNumber, name}
   └─ Return 201 Created immediately
   ↓
   ⏱️ Time: < 100ms (doesn't wait for notifications)
   ↓

3. KAFKA BROKER
   ├─ Receives message in user.events topic
   ├─ Partition 0, Offset increases
   └─ Message persisted (durable)
   ↓
   ⏱️ Time: < 10ms latency
   ↓

4. NOTIFICATION SERVICE - ROUTER
   ├─ UserEventConsumer listens to user.events
   ├─ Receives the event
   ├─ Publishes to 3 SEPARATE topics:
   │  ├─ email.events
   │  ├─ sms.events
   │  └─ push.events
   └─ All 3 published within milliseconds
   ↓
   ⏱️ Time: < 50ms total
   ↓

5. THREE INDEPENDENT CONSUMERS
   ├─ EmailNotificationConsumer (Partition 0, Offset)
   ├─ SmsNotificationConsumer (Partition 0, Offset)
   └─ PushNotificationConsumer (Partition 0, Offset)
   ↓
   All process IN PARALLEL
   ↓

6. SERVICES - ATTEMPT 1
   ├─ EmailService.sendEmail() → SUCCESS or FAIL
   ├─ SmsService.sendSms() → SUCCESS or FAIL
   └─ PushService.sendPush() → SUCCESS or FAIL
   ↓

7. RESULT:
   ├─ If SUCCESS ✓
   │  └─ Message acknowledged
   │  └─ Notification delivered
   │
   ├─ If FAIL (Attempt 1 only)
   │  └─ Schedule RETRY in 1 second
   │  └─ Message not acknowledged yet
   │
   └─ If FAIL 3+ times
      └─ Send to Dead-Letter Queue
      └─ DeadLetterQueueConsumer handles it
      └─ Alert ops team

📍 Detailed Step-by-Step Breakdown

1 Client Sends Registration Request

Where: User Service (Port 8080)

What Happens:

  • Client sends POST request with user details
  • UserController.register() method is invoked
  • Request data: id, name, email, phoneNumber (from User Service)
POST /api/users HTTP/1.1 Content-Type: application/json { "id": "user001", "name": "Alice", "email": "alice@example.com" }

Timing: ⏱️ Request received in < 1ms

2 User Service Creates User & Publishes Event

Where: User Service - UserController & UserEventProducer

Code Flow:

@PostMapping("/api/users") public ResponseEntity<User> register(@RequestBody User user) { // Step 1: Validate and store user (in-memory) // Step 2: Publish event to Kafka producer.sendUserRegistered(user); // Step 3: Immediately return response (don't wait!) return ResponseEntity.status(201).body(user); }

Event Published:

Topic: user.events Key: user001 Value: { "userId": "user001", "name": "Alice", "email": "alice@example.com", "phoneNumber": "+1-555-0000", "eventType": "USER_REGISTERED" }

Key Point: ✅ Service returns immediately (asynchronous processing)

Timing: ⏱️ < 100ms total (publication + response)

3 Client Receives 201 Response

Client receives:

HTTP/1.1 201 Created Content-Type: application/json { "id": "user001", "name": "Alice", "email": "alice@example.com" }

Important: ⚡ Client gets response BEFORE notifications are sent

User Experience: ✓ Registration completes instantly

Timing: ⏱️ < 500ms from request to response

4 Kafka Stores Message Durably

What Happens in Kafka:

  • Message stored in partition 0
  • Assigned an offset (monotonically increasing)
  • Retained based on configuration (24 hours or 1GB)
  • Available for all consumers to process
Topic: user.events ├─ Partition 0 │ ├─ Offset 0: Previous message │ ├─ Offset 1: Previous message │ └─ Offset 42: ← NEW MESSAGE (our user event) │ {userId: "user001", email: "alice@example.com", ...}

Durability: 💾 Message persisted (survives broker restart)

Timing: ⏱️ < 50ms from producer to broker

5 Notification Service Router Receives Event

Component: UserEventConsumer (the Router)

Trigger: @KafkaListener(topics = "user.events")

What Happens:

  • Consumer pool fetches new messages from partition 0
  • UserEventConsumer.handleUserRegistered() is invoked
  • Receives the event with metadata (topic, partition, offset)
  • Validates and converts event to NotificationEvent
@KafkaListener(topics = "user.events", groupId = "notification-service-group") public void handleUserRegistered( @Payload Object userEvent, @Header(KafkaHeaders.RECEIVED_TOPIC) String topic, @Header(KafkaHeaders.RECEIVED_PARTITION_ID) int partition, @Header(KafkaHeaders.OFFSET) long offset) { // Received from: user.events, partition: 0, offset: 42 logger.info("[USER-EVENT-ROUTER] Received user event"); // Convert to NotificationEvent NotificationEvent notificationEvent = convertToNotificationEvent(userEvent); }

Timing: ⏱️ < 100ms from publication to consumption

6 Router Publishes to 3 Separate Topics

The Routing Logic:

Router publishes the SAME event to 3 different topics, each with its own consumer:

routeEmailNotification(notificationEvent); // → email.events routeSmsNotification(notificationEvent); // → sms.events routePushNotification(notificationEvent); // → push.events Implementation: kafkaTemplate.send(EMAIL_TOPIC, event.getUserId(), event); kafkaTemplate.send(SMS_TOPIC, event.getUserId(), event); kafkaTemplate.send(PUSH_TOPIC, event.getUserId(), event);

Topics Created:

Topic Consumer Service Failure Rate
email.events EmailNotificationConsumer EmailNotificationService 10%
sms.events SmsNotificationConsumer SmsNotificationService 15%
push.events PushNotificationConsumer PushNotificationService 5%

Independence: ✓ Each topic has independent consumer group

Timing: ⏱️ < 50ms to publish all 3 topics

7 Three Consumers Process in Parallel

What's Happening Now:

Three independent consumers are consuming from their respective topics SIMULTANEOUSLY:

EMAIL CONSUMER SMS CONSUMER PUSH CONSUMER │ │ │ ├─ Listening to ├─ Listening to ├─ Listening to │ email.events │ sms.events │ push.events │ │ │ ├─ Consumer Group: ├─ Consumer Group: ├─ Consumer Group: │ email-notification- │ sms-notification- │ push-notification- │ consumer-group │ consumer-group │ consumer-group │ │ │ └─ When message arrives └─ When message arrives └─ When message arrives in email.events in sms.events in push.events ↓ ↓ ↓ Trigger consumer pool Trigger consumer pool Trigger consumer pool Fetch offset X Fetch offset Y Fetch offset Z

Parallel Processing: ⚡ All 3 run simultaneously

No Blocking: ✓ Each consumer independent

Timing: ⏱️ Arrival < 50ms

8 Services Execute (Attempt 1)

All Three Attempt in Parallel:

ATTEMPT 1: EmailNotificationConsumer: └─ emailService.sendEmail(event) ├─ Extract email address ├─ Attempt sending via SendGrid/SES └─ SUCCESS ✓ or FAIL ✗ SmsNotificationConsumer: └─ smsService.sendSms(event) ├─ Extract phone number ├─ Attempt sending via Twilio/SNS └─ SUCCESS ✓ or FAIL ✗ PushNotificationConsumer: └─ pushService.sendPush(event) ├─ Extract user ID ├─ Attempt sending via Firebase/OneSignal └─ SUCCESS ✓ or FAIL ✗

Probability (on each service):

  • Email: 90% success, 10% fail
  • SMS: 85% success, 15% fail
  • Push: 95% success, 5% fail

Overall Success Rate: 90% × 85% × 95% ≈ 72.6% all succeed on first try

Timing: ⏱️ 50-500ms per service

9 Message Acknowledgment (Success Case)

When Service Succeeds:

// In EmailNotificationConsumer: try { emailService.sendEmail(event); // SUCCESS - message automatically acknowledged // Consumer group offset advances // Message no longer needs to be processed logger.info("✓ Email sent successfully"); } catch (EmailNotificationException e) { // FAIL - message NOT acknowledged yet // Framework will retry }

Kafka Behavior:

  • After method completes without exception
  • Kafka automatically acknowledges the message
  • Consumer group offset moves forward
  • Message won't be reprocessed

Timing: ⏱️ Acknowledged automatically after success

⏱️ Timing Analysis

Component Latencies

Component Operation Typical Latency
User Service Process request + publish event 50-100ms
Kafka Broker Store message 10-50ms
Consumer Pool Fetch and process message 50-100ms
Router Convert + publish to 3 topics 30-50ms
Email Service Send email (success) 200-500ms
SMS Service Send SMS (success) 100-300ms
Push Service Send push (success) 50-150ms

End-to-End Timing

Scenario 1: All Success (Happy Path)

Client sends request ↓ 50-100ms User Service creates + publishes ↓ 10-50ms Kafka stores message ↓ 50-100ms Router consumes and routes ↓ 30-50ms 3 consumers triggered (in parallel) ↓ 200-500ms (longest service = bottleneck) All 3 notifications delivered TOTAL: ~400-800ms for all notifications (Client gets response in ~60-100ms)

Scenario 2: One Retry (Email Fails Once)

Attempt 1 fails @ 300ms Schedule retry @ 300ms ↓ 1000ms (1 second delay) Attempt 2 @ 1300ms Attempt 2 succeeds @ 1500ms SMS & Push still deliver @ 300-500ms TOTAL: ~1500ms for email, ~500ms for others (SMS & Push don't wait for email)

Scenario 3: All Retries Fail (DLQ)

Attempt 1 fails @ 300ms Attempt 2 fails @ 1300ms (after 1s) Attempt 3 fails @ 3300ms (after 2s) Send to DLQ @ 3300ms DLQ consumer processes @ 3310ms TOTAL: ~3.3 seconds until DLQ (Then manual retry as needed)

Critical Path Analysis

Critical Path (Determines Overall Latency):

1. User Service: 50-100ms
2. Kafka Publish: 10-50ms
3. Router Consume & Route: 50-100ms
4. Slowest Service (Email usually): 200-500ms
                 ↓
         CRITICAL: 310-750ms

Non-Critical Paths (Parallel):
├─ SMS Service: 100-300ms (can be slower than email)
└─ Push Service: 50-150ms (fastest)

Bottleneck: Whichever service is slowest!
Current: Email service (200-500ms)

Optimization: Parallelize within services
(Use thread pools, async I/O, etc.)

│ 1. Engineer investigates       │
│ 2. Finds root cause            │
│ 3. Fixes underlying issue      │
│ 4. Manually retries from DB    │

OUTCOMES:
═════════════════════════════════════════════════════════════════
✅ HAPPY PATH (All Success):          ~350-800ms all delivered
🔄 RETRY PATH (One Fails Then Works):  ~1-3s for that channel
💀 DLQ PATH (All Fail):                ~3.3s to DLQ, manual retry after

KEY BENEFIT: Each channel independent!
             Email down ≠ SMS down ≠ Push down

Data Structure Flow

STEP 1: User registers Input: {id: "user001", name: "Alice", email: "alice@example.com"} STEP 2: User Service publishes Topic: user.events Message: {id: "user001", name: "Alice", email: "alice@example.com"} STEP 3: Router converts Target: NotificationEvent { userId: "user001", name: "Alice", email: "alice@example.com", phoneNumber: "+1-555-0000", eventType: "USER_REGISTERED" } STEP 4: Router publishes to 3 topics Topic: email.events Message: NotificationEvent { ... } Topic: sms.events Message: NotificationEvent { ... } Topic: push.events Message: NotificationEvent { ... } STEP 5: Consumers process independently Each receives same NotificationEvent Each extracts relevant field: - Email consumer: event.getEmail() - SMS consumer: event.getPhoneNumber() - Push consumer: event.getUserId() STEP 6: Success or Failure Success: Message acknowledged, offset advances Failure: Scheduled for retry, offset stays same STEP 7: All Retries Fail Message sent to DLQ with metadata: { originalMessage: NotificationEvent { ... }, errorMessage: "Connection timeout", attempt: 3, originalTopic: "email.events", originalPartition: 0, originalOffset: 10 }
