📖 Project Overview

NotifyHub is a distributed microservices application that demonstrates asynchronous event-driven communication between two independent Spring Boot services. The system uses Apache Kafka as a message broker to enable loose coupling and reliable event processing across services.

The project consists of two main microservices:

👤 User Service

Port: 8080

Handles user registration and publishes user registration events to Kafka for other services to consume.

🔔 Notification Service

Port: 8090

Listens to user events from Kafka and simulates sending notifications (emails/SMS) to users.

🎯 What We're Trying to Achieve

This project demonstrates key concepts in modern distributed system architecture:

  1. Microservices Architecture: Breaking down a monolithic application into independent, deployable services that handle specific business domains.
  2. Event-Driven Communication: Using Kafka to enable services to communicate through events rather than direct API calls, reducing tight coupling.
  3. Asynchronous Processing: Services don't need to wait for responses. The user service publishes an event and immediately returns a response, while the notification service processes it in the background.
  4. Scalability: Services can be scaled independently. You can run multiple instances of the notification service to handle high event volumes.
  5. Fault Tolerance: If the notification service is temporarily down, messages are retained in Kafka and processed when it comes back online.
  6. Spring Boot Best Practices: Using Spring Boot, Spring Kafka, and Maven for building production-ready applications.
Real-World Use Case: This pattern is used in systems like ride-sharing apps (Uber), e-commerce platforms (Amazon), and social media networks (Twitter) where user actions trigger notifications that must be delivered reliably without blocking the main user flow.

🏗️ System Architecture

NotifyHub follows an event-driven microservices architecture with message-oriented communication: 

┌─────────────────────────────────────────────────────────────────┐
│                         CLIENT APPLICATION                       │
└────────────────┬────────────────────────────────────┬────────────┘
                 │                                     │
        POST /api/users                      POST /api/events

                 │                                     │
        ┌────────▼─────────┐                  ┌──────▼───────────┐
        │                  │                  │                  │

        │  (Port 8080)     │                  │  SERVICE         │
        │                  │                  │  (Port 8090)     │
        │ ┌──────────────┐ │                  │                  │

        │ └──────┬───────┘ │                  │ │ EventCtrlr   │ │

            │  In-Memory   │          │  In-Memory      │

FLOW:
1. User registers via POST /api/users
2. User Service publishes UserRegisteredEvent to Kafka
3. Notification Service (Consumer) receives event instantly
4. Notification Service simulates sending notification

Data Flow:

  1. User Registration Request: Client sends user details (id, name, email) to User Service
  2. Event Publishing: User Service immediately publishes a "UserRegisteredEvent" to Kafka topic user.events
  3. Event Consumption: Notification Service consumes the event from Kafka asynchronously
  4. Notification Delivery: Notification Service simulates sending an email/SMS notification
  5. Response to Client: User Service responds with 201 status immediately (doesn't wait for notification)

🚀 Step-by-Step Setup Instructions

1

Navigate to Project Directory

Open your terminal/command prompt and navigate to the project root directory:

cd "path/to/NotifyHub - Distributed Notification Service"

On Windows: The path should be:

cd "C:\Users\Prem\OneDrive\Desktop\NotifyHub – Distributed Notification Service\NotifyHub - Distributed Notification Service"
2

Start Kafka and Zookeeper with Docker Compose

This step starts both Kafka and Zookeeper containers that are required for message brokering:

docker-compose up -d

What this does:

  • -d flag runs containers in detached mode (background)
  • Starts Zookeeper container on port 2181
  • Starts Kafka broker on port 9092
  • Creates the Docker network for service communication
✓ Success Indicator: You should see messages like:
Creating zookeeper ... done
Creating kafka ... done

Verify Kafka is running:

docker-compose ps
3

Build User Service

Navigate to the user-service directory and build it with Maven:

cd user-service
mvn clean install

What this does:

  • clean - Removes any previous build artifacts
  • install - Compiles source code, runs tests, and packages as JAR
  • Downloads all required dependencies from Maven Central Repository

Build Time: First build may take 5-10 minutes (downloading dependencies)

✓ Success Indicator: You should see:
BUILD SUCCESS
4

Run User Service

Start the User Service application:

mvn spring-boot:run

Expected Output: You should see Spring Boot startup messages including:

Started UserServiceApplication in X.XXX seconds (JVM running for Y.YYY)
Tomcat started on port(s): 8080
ℹ️ Keep this terminal running! The service will continue running in this terminal. Do NOT close it. You'll need to open a new terminal for the next step.
5

Build Notification Service (New Terminal)

Open a NEW terminal/command prompt and navigate to the notification-service:

cd "path/to/project"
cd notification-service
mvn clean install
✓ Success Indicator: You should see BUILD SUCCESS message
6

Run Notification Service

In the same terminal (notification-service directory), run the service:

mvn spring-boot:run

Expected Output:

Started NotificationServiceApplication in X.XXX seconds (JVM running for Y.YYY)
Tomcat started on port(s): 8090
✓ System is Now Ready! You have successfully started:
  • ✓ Zookeeper (Port 2181)
  • ✓ Kafka Broker (Port 9092)
  • ✓ User Service (Port 8080)
  • ✓ Notification Service (Port 8090)

Terminal Layout:

After all steps, you should have three running processes:

Terminal 1: [RUNNING] docker-compose up -d → Kafka + Zookeeper
Terminal 2: [RUNNING] mvn spring-boot:run → User Service (Port 8080)
Terminal 3: [RUNNING] mvn spring-boot:run → Notification Service (Port 8090)

🔌 API Endpoints & Testing

Once both services are running, you can test them using the following endpoints:

Endpoint 1: Register a User

POST http://localhost:8080/api/users

Purpose: Register a new user and publish a registration event to Kafka

Request Headers:

Header Value
Content-Type application/json

Request Body:

{ "id": "user123", "name": "Alice Johnson", "email": "alice@example.com" }

Expected Response (HTTP 201 Created):

{ "id": "user123", "name": "Alice Johnson", "email": "alice@example.com" }

How to Test with curl:

curl -X POST http://localhost:8080/api/users \
-H "Content-Type: application/json" \
-d '{
"id": "user123",
"name": "Alice Johnson",
"email": "alice@example.com"
}'

How to Test with Postman:

  1. Open Postman application
  2. Click "New" → "Request"
  3. Set method to POST
  4. Enter URL: http://localhost:8080/api/users
  5. Go to "Body" tab → Select "raw" → Select "JSON"
  6. Paste the JSON request body above
  7. Click "Send"

Endpoint 2: Publish Custom Event

POST http://localhost:8090/api/events

Purpose: Publish a custom event (for future extension)

Request Body:

{ "type": "ORDER_PLACED", "payload": { "orderId": "ORD-123", "amount": 99.99 } }

Expected Response (HTTP 202 Accepted):

{ "type": "ORDER_PLACED", "payload": { "orderId": "ORD-123", "amount": 99.99 } }

What Happens After Each Request:

When you register a user via the User Service:

  1. User Service receives the request immediately
  2. User Service publishes a "UserRegisteredEvent" to Kafka topic user.events
  3. User Service responds with 201 status code immediately (doesn't wait)
  4. Notification Service receives the event from Kafka almost instantly
  5. Notification Service prints the event to console (simulating notification delivery)

Observe Console Output:

User Service Console (Terminal 2): Will show request details

Notification Service Console (Terminal 3): Will show the event received:

[NotificationService] Consumed user event: {...}
[NotificationService] Simulating notification delivery (email/SMS) for user event
💡 Key Observation: Notice how the User Service responds immediately, while the Notification Service processes the event asynchronously. This is the power of event-driven architecture!

🔧 Troubleshooting Guide

Issue: Docker containers fail to start

Problem: docker-compose up -d fails or containers don't start

Solutions:

  1. Ensure Docker daemon is running
    docker --version
  2. Check if ports 2181 and 9092 are already in use
    docker-compose ps
  3. Stop conflicting containers
    docker stop $(docker ps -aq)
  4. Check logs
    docker-compose logs

Issue: "Port already in use" errors

Problem: Error message says port 8080, 8090, or 9092 is already in use

Solution: Kill the process using that port or change the port in application.yml

For Windows:

netstat -ano | findstr :8080
taskkill /PID <PID> /F

For Mac/Linux:

lsof -i :8080
kill -9 <PID>

Issue: Maven build fails with "No Java Runtime"

Problem: Could not find or load main class or Java version issues

Solutions:

  1. Verify Java installation
    java -version
  2. Set JAVA_HOME environment variable:

    Windows (Command Prompt as Admin):

    setx JAVA_HOME "C:\Program Files\Java\jdk-17"

    Mac/Linux:

    export JAVA_HOME=$(/usr/libexec/java_home)
  3. Restart terminal and try building again

Issue: Services can't connect to Kafka

Problem: Services fail to start with "Failed to connect to Kafka"

Solutions:

  1. Verify Kafka is running
    docker-compose ps
  2. Wait 30 seconds for Kafka to fully start (it needs time to initialize)
  3. Check Kafka logs
    docker-compose logs kafka
  4. Verify bootstrap server address in application.yml is localhost:9092

Issue: Events not being received by Notification Service

Problem: User Service publishes events but Notification Service doesn't show them

Solutions:

  1. Check both services are running on correct ports (8080 and 8090)
  2. Ensure topic user.events exists in Kafka
  3. Check consumer group configuration is correct
  4. Check logs for serialization errors (JSON parsing issues)
  5. Restart both services in correct order: User Service first, then Notification Service

Issue: Build takes too long or fails with network errors

Problem: Maven can't download dependencies

Solutions:

  1. Check internet connection
  2. Try clearing Maven cache
    mvn clean
  3. Force update dependencies
    mvn clean install -U
  4. Check Maven settings are correct
    mvn -v

Issue: Need to stop everything and restart

To cleanly shutdown:

  1. In each running terminal, press Ctrl + C to stop the service
  2. Stop Docker containers
    docker-compose down
  3. To start fresh, follow the setup steps again

notification-service/
├── pom.xml # Maven configuration and dependencies
└── src/
└── main/

│ ├── controller/

│ │ └── UserEventConsumer.java # Consumes events from Kafka

Key Files Explained:

docker-compose.yml

Purpose: Defines the infrastructure (Kafka + Zookeeper) needed for the application

Contains:

  • Zookeeper service (metadata management for Kafka)
  • Kafka broker service (message broker)
  • Port mappings and environment variables

pom.xml

Purpose: Maven configuration file for each service

Defines:

  • Project metadata (name, version, group ID)
  • Java version (17)
  • Spring Boot version and dependencies
  • Spring Kafka dependency for Kafka integration
  • Jackson for JSON serialization
  • Lombok for code generation

*Application.java

Purpose: Entry point for Spring Boot application

What it does:

  • Marks the class with @SpringBootApplication
  • Runs SpringApplication.run() to start the embedded Tomcat server
  • Initializes Spring context and auto-wires all beans

Example: UserServiceApplication starts User Service on port 8080

UserController.java

Purpose: Handles HTTP requests for user registration

Key Methods:

  • @PostMapping - Accepts POST requests to /api/users
  • register() - Receives User object, publishes event, returns 201 response

UserEventProducer.java

Purpose: Publishes user registration events to Kafka

Key Methods:

  • sendUserRegistered() - Creates UserRegisteredEvent and sends to Kafka topic "user.events"
  • Uses KafkaTemplate for sending messages

UserEventConsumer.java

Purpose: Listens to and processes user events from Kafka

Key Methods:

  • @KafkaListener - Subscribes to "user.events" topic
  • handleUserRegistered() - Receives event and simulates notification delivery

KafkaConfig.java (User Service)

Purpose: Configures Kafka producer

Configures:

  • Bootstrap servers (localhost:9092)
  • Key and value serializers (StringSerializer, JsonSerializer)
  • ProducerFactory and KafkaTemplate beans

KafkaConfig.java (Notification Service)

Purpose: Configures Kafka consumer

Configures:

  • Bootstrap servers (localhost:9092)
  • Consumer group ID (notification-service-group)
  • Key and value deserializers (StringDeserializer, JsonDeserializer)
  • ConcurrentKafkaListenerContainerFactory bean

application.yml

Purpose: Runtime configuration for each service

User Service (application.yml):

server:
port: 8080 # Service runs on this port
spring:
kafka:
bootstrap-servers: localhost:9092 # Kafka location

Notification Service (application.yml):

server:
port: 8090
spring:
kafka:
consumer:
bootstrap-servers: localhost:9092
group-id: notification-service-group
